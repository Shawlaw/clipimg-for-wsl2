# clipImg Rust 单 EXE 方案实施计划

## Context

当前 clipImg 由 PowerShell 守护进程 + AutoHotkey 热键脚本两部分组成，需要分别启动、分别维护。目标是合并为一个 Rust 单 EXE，支持可配置热键和路径，体积控制在 5MB 以内。

## 技术选型

| 组件 | Crate | 说明 |
|------|-------|------|
| 剪贴板图片读取 | `arboard` 3.x | 1Password 维护，直接输出 RGBA 像素数据，无需手写 Win32 FFI |
| 图片编码保存 | `image` 0.25 | RGBA → PNG 编码 |
| 全局热键 | `global-hotkey` 0.6 | Tauri 出品，支持字符串解析热键（`"Alt+Insert"`），运行时可重注册 |
| 系统托盘 | `tray-icon` 0.21 + `muda` 0.17 | Tauri 出品，右键菜单+tooltip |
| 事件循环 | `tao` 0.3 | Tauri 对 winit 的 fork，与 global-hotkey/tray-icon 深度集成 |
| 键盘模拟 / 多格式剪贴板 | `windows-sys` | 方案 A: SendInput + KEYEVENTF_UNICODE + IME 切换；方案 C: 多格式剪贴板 |
| 配置文件 | `serde` + `serde_json` | JSON 格式 |
| 日志 | `log` + `env_logger` | 标准方案 |
| 哈希去重 | `md-5` crate | MD5 哈希比对（与现有 PS 脚本一致） |

交叉编译方案：`cargo-xwin` + `x86_64-pc-windows-msvc` 目标（从 WSL2 构建原生 Windows EXE）

## 配置文件格式 (`config.json`，放在 EXE 同目录）

```json
{
  "hotkey": "Alt+Insert",
  "output_path": "/workspace/.clip/latest.png",
  "save_dir": ".clip",
  "poll_interval_ms": 800,
  "max_history_days": 7
}
```

- `hotkey`: 支持格式如 `"Alt+Insert"`, `"Ctrl+Shift+V"`, `"Super+V"` 等（由 global-hotkey 的 `HotKey::try_from()` 解析）。**空字符串 `""` 表示不注册热键，使用多格式剪贴板模式（方案 C）**
- `output_path`: 热键触发时输入到终端的**容器侧路径**（如 `/workspace/.clip/latest.png`）
- `save_dir`: 图片在 **Windows 侧**的保存目录。支持两种形式：
  - **相对路径**（如 `.clip`）：相对于 workspace 根目录，workspace 通过 EXE 所在位置向上推导（与 PS 脚本一致：`clipImg/clipimg-app/` → `clipImg/` → workspace root）
  - **绝对路径**（如 `E:\WorkingProjects\workspace\.clip`）：直接使用，不推导
- `poll_interval_ms`: 剪贴板轮询间隔
- `max_history_days`: 历史图片最大保留天数（clean 功能）

两个路径的关系：`save_dir` 是 Windows 文件系统上的实际写入位置，`output_path` 是 Docker 容器内能识别的路径，两者通过目录挂载映射到同一个物理文件。

## 核心逻辑

### 1. 剪贴板监控 (`clipboard.rs`)

移植现有 `clipboard-watcher.ps1` 的去重逻辑：
1. 轮询 `arboard::Clipboard::get_image()` 获取 RGBA 数据
2. 用 `image` crate 编码为 PNG，先写 `_tmp_clip.png`
3. 两级去重：文件大小 → MD5 哈希（与 PS 脚本完全一致）
4. 新图片：`_tmp_clip.png` → `clip_YYYYMMDD_HHmmss.png` + 覆盖 `latest.png`
5. 旧图片：删除 `_tmp_clip.png`

### 2. 路径输入 (`input.rs`) — 双模式

**模式 C（hotkey 为空）**：
截图保存后，自动设置多格式剪贴板：
1. CF_BITMAP / CF_DIB：截图 PNG 数据
2. CF_UNICODETEXT：`output_path` 字符串
3. CF_HDROP：Windows 侧 latest.png 路径
用户在终端 Ctrl+V / Shift+Insert → 得到路径

**模式 A（hotkey 有值）**：
热键触发时：
1. 保存当前 IME 布局
2. 切换到英文布局 (0x0409)
3. 逐字符调用 SendInput（KEYEVENTF_UNICODE，key-down 和 key-up 分开调用）
4. 恢复 IME 布局

### 3. 托盘菜单 (`main.rs`)

右键菜单项：
- **状态**（灰色文字）："运行中 / 监控目录: .clip"
- **打开配置文件**：用默认编辑器打开 `config.json`
- **打开图片目录**：用资源管理器打开 `save_dir/`
- **开机自启**（Checkbox）：读写注册表 `HKCU\...\Run`
- **退出**

### 4. 事件循环

```
main() → tao 事件循环
  ├── clipboard_timer (每 poll_interval_ms 触发) → clipboard.rs
  ├── GlobalHotKeyEvent::receiver() → input.rs
  ├── MenuEvent::receiver() → 托盘菜单处理
  └── TrayIconEvent::receiver() → 托盘点击
```

## v2.1 路径输入方案重构（2026-04-08）

### 背景

v2.0 实现后，路径输入方案经历了多次迭代但都有问题：
1. **SendInput + KEYEVENTF_UNICODE 批量发送**：日志显示成功但无文字输出（原因：key-down/key-up 在同一 SendInput 调用中，部分应用/IME 下丢失）
2. **SendInput + VkKeyScanW**：终端可用但很慢，普通输入框因 IME 拦截不生效
3. **剪贴板 + Ctrl+V**：最通用但破坏用户剪贴板内容

### 参考项目研究

**Nailuu/wsl-screenshot-cli**（Go + PowerShell）：
- 不做键盘模拟，改用**多格式剪贴板**（CF_BITMAP + CF_UNICODETEXT + CF_HDROP 三种格式共存）
- 用户自己在终端 Ctrl+V / Shift+Insert 粘贴 → 得到 CF_UNICODETEXT（文件路径）
- 在图片应用粘贴 → 得到 CF_BITMAP（图片）
- 在资源管理器粘贴 → 得到 CF_HDROP（文件）
- 用"三种格式同时存在"作为指纹，区分自己的写入 vs 用户新截图
- **局限**：依赖用户手动粘贴，无法自动输入

**unclejimao/WSL-Image-Clipboard-Helper**（AutoHotkey v2 + PowerShell）：
- 按 Alt+V 自动输入路径，不碰剪贴板
- 核心流程：保存当前 IME → `PostMessage(WM_INPUTLANGCHANGEREQUEST)` 切英文 → AHK `SendText()` → 等 500ms → 恢复 IME
- 启动时预加载英文布局 `LoadKeyboardLayoutW("00000409")`，缓存句柄
- AHK `SendText()` 内部是 SendInput + KEYEVENTF_UNICODE，逐字符处理
- **关键发现**：之前的 SendInput 失败很可能是两个原因——(1) 批量发送 bug (2) 未切换 IME

### 决策：混合方案（方案 C + A）

根据 `config.json` 中 `hotkey` 字段是否为空，自动选择模式：

**模式 C（无热键）**：`"hotkey": ""`
- 不注册全局热键
- 截图保存后自动设置多格式剪贴板（CF_BITMAP + CF_UNICODETEXT + CF_HDROP）
- 用户自己 Ctrl+V / Shift+Insert 粘贴，终端得到文件路径
- 用 Nailuu 的"三格式指纹"判断剪贴板是否为自己的增强写入

**模式 A（有热键）**：`"hotkey": "Alt+Insert"`
- 注册全局热键
- 热键触发时走 SendInput + KEYEVENTF_UNICODE + IME 临时切换
- 逐字符分开调用 SendInput（修复批量发送 bug）
- 发送前切英文 IME，发送后恢复
- 不碰剪贴板

### 技术实现要点

**模式 C（多格式剪贴板）**：
- 使用 `arboard::Clipboard` 的底层 API 或 Windows `SetClipboardData` 设置多格式
- `CF_UNICODETEXT`：output_path 字符串（如 `/workspace/.clip/latest.png`）
- `CF_BITMAP` / `CF_DIB`：截图的 PNG 图片数据
- `CF_HDROP`：Windows 侧 latest.png 文件路径
- 剪贴板指纹检测：image + text + file drop 同时存在 = 跳过

**模式 A（SendInput + IME 切换）**：
- 启动时：`LoadKeyboardLayoutW("00000409", KLF_ACTIVATE)` 预加载英文布局
- 热键触发时：
  1. `GetWindowThreadProcessId(GetForegroundWindow())` → 线程 ID
  2. `GetKeyboardLayout(threadId)` → 保存当前布局
  3. `PostMessageW(HWND_BROADCAST, WM_INPUTLANGCHANGEREQUEST, 0, eng_hkl)` → 切英文
  4. `Sleep(100ms)` → 等待切换生效
  5. 逐字符调用 `SendInput`（每次 1 个 key-down + 1 个 key-up，分开调用）
  6. `Sleep(100ms)` → 等待输入完成
  7. `PostMessageW(HWND_BROADCAST, WM_INPUTLANGCHANGEREQUEST, 0, prev_hkl)` → 恢复
- 需要的 Win32 API：`LoadKeyboardLayoutW`, `GetKeyboardLayout`, `GetWindowThreadProcessId`, `GetForegroundWindow`, `PostMessageW`, `SendInput`
- `windows-sys` features 需新增：`Win32_UI_Input_KeyboardAndMouse`（已有）, `Win32_UI_TextManagement`（IME）, 或直接用 `user32` 链接

### 涉及文件改动

| 文件 | 改动 |
|------|------|
| `config.rs` | `hotkey` 允许空字符串，`validate()` 跳过空 hotkey 校验 |
| `input.rs` | 重写：新增 `set_multi_format_clipboard()`（模式 C）和 `send_text_with_ime()`（模式 A） |
| `clipboard.rs` | `poll()` 返回值改为 `PollResult`，携带是否有新图片信息 |
| `main.rs` | 条件注册热键；新图片保存后若模式 C 则调用多格式剪贴板设置 |
| `Cargo.toml` | `windows-sys` features 新增 IME 相关 |

## 风险与应对

| 风险 | 应对 |
|------|------|
| `arboard` 返回的图片数据与 PS 脚本不一致 | 验证 RGBA→PNG 输出与 PS 的 `GetImage().Save()` 输出一致 |
| `cargo-xwin` 交叉编译链路出问题 | 备选：直接在 Windows PowerShell 里 `cargo build --release` |
| `global-hotkey` 与某些应用冲突 | 提供配置文件让用户换快捷键 |
| SendInput 在 UAC 提权窗口无效 | 这是 Windows 安全限制，无法绕过，在 README 注明 |
| arboard 不支持多格式剪贴板 | 备选：用 `windows-sys` 直接调用 Win32 `SetClipboardData` |
| IME 切换在某些应用不生效 | 已验证 unclejimao 方案可行；极端情况可降级到模式 C |
