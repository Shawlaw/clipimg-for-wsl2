# v1.0.12 计划

## 变更内容

### 1. 托盘菜单点击响应优化

**现状**：消息循环每次 `GetMessageW` 只取一条消息，处理完所有事件后才进入下一轮。muda 的托盘菜单点击通过 `WM_COMMAND` → 窗口过程 → crossbeam channel 传递，如果队列中有其他消息积压，菜单事件需要多轮循环才能被 `try_recv()` 取到，表现为点击后响应延迟。

**改动**：
- `DispatchMessageW` 之后加 `PeekMessageW` 循环排空消息队列
- 一轮循环内所有挂起的窗口消息都被 Dispatch，muda 事件能立即被 `try_recv()` 取到
- `GetMessageW` 仍然用于首次阻塞等待，空闲时零 CPU
- 新增 `PeekMessageW`、`PM_REMOVE`、`WM_QUIT` 的 import
- `WM_QUIT` 在 PeekMessageW 循环中检测到时重新 PostQuitMessage，留给下一轮 GetMessageW 处理

**注意**：PeekMessageW 会取走所有线程消息（包括 `wm_clip_changed`），但这些是 `PostThreadMessageW` 投递的无窗口消息，`DispatchMessageW` 对它们是空操作。当前不会造成问题——自定义消息在主循环的 `GetMessageW` 阶段已被处理。需要在 Windows 上实测确认点击响应确实改善。

### 2. Debug 构建模式（调测便利性）

**现状**：每次测试新版本需要手动退出当前运行的 exe → 改名 → 复制新 exe → 运行，流程繁琐。

**方案**：新增 `debug_build` feature，编译出 `clipimg_debug.exe`。启动时自动终止 release 版本进程，实现"复制过去双击即测"。

**改动**：

#### Cargo.toml
- 新增 `[features]` 表，加入 `debug_build`
- 新增 `[[bin]]` target：`name = "clipimg_debug"`，`path = "src/main.rs"`，`required-features = ["debug_build"]`

#### main.rs — 启动时终止 release 进程
```rust
#[cfg(feature = "debug_build")]
{
    // 终止正在运行的 clipimg.exe（release 版本）
    let _ = std::process::Command::new("taskkill")
        .args(["/IM", "clipimg.exe", "/F"])
        .output();
    std::thread::sleep(std::time::Duration::from_millis(500));
}
```

#### main.rs — 互斥体名称
- debug 版本使用 `Global\clipimg_debug`，避免与 release 版冲突（虽然已 kill，但加保险）
- 通过 `cfg!` 宏选择 mutex 名称

#### main.rs — 托盘 tooltip 标记
- debug 版本在 tooltip 和状态菜单中显示 `(debug)` 标记，方便区分

#### 构建命令
- 本地调试：`cargo xwin build --target x86_64-pc-windows-msvc --features debug_build --release --bin clipimg_debug`
- Release 构建不变：`cargo xwin build --target x86_64-pc-windows-msvc --release`（不带 feature）
- GitHub Actions 无需修改，只编译 release 版本

## 不做的事

- 不做自动下载更新（超出范围）
- 不改 CI 流程
- debug 版本不做热重载

## 验证

- `cargo xwin check --target x86_64-pc-windows-msvc` 编译通过（不带 feature）
- `cargo xwin check --target x86_64-pc-windows-msvc --features debug_build --bin clipimg_debug` 编译通过
- Windows 实测：debug exe 启动后自动终止 release 进程，托盘显示 `(debug)` 标记
- Windows 实测：托盘菜单点击响应速度改善
